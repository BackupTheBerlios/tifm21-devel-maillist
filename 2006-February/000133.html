On Tue, Feb 14, 2006 at 07:36:46AM -0800, Alex Dubov wrote:
> Thanks for the info. For some reason I can't contact
> list.drzeus.cx (some network problem). I, however,
> figured the solution from TI's implementors guide:
> setpci -s 06:09.0 86.b=90:d0
> (that is toggle bit 6 of register 86; bit 2 of
> register 4c is supposed to disable MMC/SD flashmedia
> interface). I had to force bus rescan using fakephp
> too.

Interesting. All the hardware I've seen with the controller has had the 
sdhci function visible at all times. What sort of machine do you have?
 
> Does anybody know what benefits can I get from
> integration with MMC layer? Yes, it implements MMC
> access state machine and common primitives, but those
> hardly account for 20% of my MMC reveng code. I'm even
> assuming that SDHCI will do a better job of accessing
> SD cards, but it can not serve as a base for MS or xD
> support, can it?

Using the generic mmc code means that any improvements in there 
automatically apply to the TI driver - there's recently been code added 
for MMCv4 support, and there are some patches for supporting the MMC 
password specification. However, as you suggest it doesn't provide 
anything terribly useful for MS or xD. Ideally we'd have separate 
protocol stacks for those as well, just in case anyone figures out how 
to drive any other controllers, but I can imagine that that's not a 
short-term goal.
